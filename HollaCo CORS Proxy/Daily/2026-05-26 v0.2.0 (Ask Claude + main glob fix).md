# v0.2.0 — POST /api/claude route

Meaningful minor release of [[../Projects/2026-05-23 HollaCo CORS Proxy|HollaCo CORS Proxy]] supporting HollaCo Command Center [[../../../hollaco-command-center/HollaCo Command/Daily/2026-05-26 Phase 7.0 (v1.6.0-1.6.2)|Phase 7.0 (v1.6.x)]]. New `POST /api/claude` route alongside the existing GET `/api/proxy`. Forwards prompt + widgetState to Anthropic Messages API with the key from Function App Application Settings. Enforces per-IP + global rate limits in-memory.

#project/cors-proxy #topic/ai #topic/anthropic #topic/rate-limit #topic/azure-functions #lessons/main-glob

## What shipped

- `src/rateLimit.js`: pure `checkRateLimit(ip, now)` over module-level `Map<ip, timestamps[]>` + `globalTimestamps[]`. Per-IP cap 60/min sliding window; global cap 1000/day rolling window. State resets on cold start (acceptable at trio scale per spec).
- `src/functions/claude.js`: HTTP-triggered POST handler at `/api/claude`. Validates `prompt` (≤4000 chars) + `widgetState` (≤20000 chars), runs through rate-limit check, calls Anthropic Messages API server-side with `x-api-key` header, returns `{text, model, usage}` on success. Maps Anthropic 4xx → 502, 5xx/timeout → 504. CORS headers on every response.
- Two new Function App Application Settings (one-time setup by Ross):
  - `ANTHROPIC_API_KEY` — secret, never enters git
  - `ANTHROPIC_MODEL` — `claude-sonnet-4-6`
- `package.json` main field changed from `"src/functions/proxy.js"` to `"src/functions/*.js"` (glob) so Azure Functions v4 auto-discovers all function modules. See "The main glob bug" section below.

18 new tests (6 rate-limit + 12 handler). Total 10 → **28**. Auto-deployed via OIDC on push to main. Smoke verified.

## The main glob bug — first deploy returned 404 on /api/claude

This is the lesson worth recording verbatim.

### What happened

After the initial v0.2.0 push, smoke test of the new route failed:

```
$ curl -X POST https://proxy.hollaco.com/api/claude -d '{}'
HTTP/1.1 404 Not Found
Content-Length: 0
Server: Kestrel
```

The existing `GET /api/proxy` still worked fine. The deploy log said "Successfully deployed web package to Function App". The function file `src/functions/claude.js` was definitely in the deploy bundle (verified). But Azure didn't know about the route.

### Why it happened

`package.json` had `"main": "src/functions/proxy.js"` — set when the proxy was a single-function repo back at v0.1.0. Azure Functions v4 uses the `main` field to determine which module to import at runtime. With a single file, it imports that file, which runs the file's `app.http(...)` registration once. **It does NOT scan `src/functions/` for additional modules.**

So `src/functions/claude.js` shipped to Azure but was never imported, so its `app.http('claude', ...)` registration never ran, so the route was never registered. The 404 was Azure correctly saying "no function registered at this route".

### What we fixed it with

Azure Functions v4 supports glob patterns in `main`:

```diff
- "main": "src/functions/proxy.js",
+ "main": "src/functions/*.js",
```

After this change, Azure loads all `.js` files under `src/functions/` at startup, each runs its `app.http(...)` registration, all routes get registered. Both `/api/proxy` and `/api/claude` work.

### Lesson

**When adding the second HTTP-triggered function to an Azure Functions v4 repo, change `package.json#main` from a single-file path to a glob.** Or use a barrel file (`src/index.js`) that imports each function module. The single-file `main` works for one-function apps but silently breaks the second function. Worth a follow-up: write a one-line README/AGENTS note for any agentic worker adding a third function in the future — "verify main is a glob pattern, otherwise your new function will return 404".

## Function App Platform CORS quirk (widget-side fix)

Adjacent story worth recording even though the fix lives in the widget: the Function App's Platform CORS layer intercepts OPTIONS preflight requests BEFORE our function handler runs. When the preflight includes `Access-Control-Request-Headers: content-type` (which it does for any `Content-Type: application/json` POST), the Platform CORS layer 204s the response but does NOT echo `Access-Control-Allow-Origin` or `Access-Control-Allow-Headers`. The browser then rejects the preflight and the actual POST never fires.

Tried both `*` and empty allowed-origins in the Function App CORS panel — same result. The Platform CORS layer fundamentally doesn't handle preflights with custom request headers.

**Widget-side fix** (in `hollaco-command-center/src/lib/ai.js`): drop the `Content-Type: application/json` header so the POST qualifies as a CORS "simple request" — no preflight triggered. The proxy's `req.json()` parses the body regardless of Content-Type, so end-to-end behavior is unchanged.

**Implication for future proxy POST routes:** widget callers should NOT add custom request headers when calling proxy POST routes unless we accept that those callers will fail until we work around Platform CORS another way. Document this in any future POST route docs.

## Routes (post-v0.2.0)

| Method | Route | Purpose | Auth |
|---|---|---|---|
| GET | `/api/proxy?upstream=<key>` | Re-serve CORS-blocked Statuspage/health endpoints (allowlist in `src/upstreams.js`) | Allowlist only |
| POST | `/api/claude` | Forward prompt + widgetState to Anthropic Messages API; rate-limited (60/min per IP + 1000/day global) | Anonymous + rate limit |

## Why the POST route lives here (vs. separate Function App)

Same domain (`proxy.hollaco.com`) means widget only configures one URL. Same OIDC deploy means no new Entra app or federated credential. Same repo means one PR review surface. Trade-off: the proxy is no longer "just a CORS re-server" — it's an "HTTP gateway for things the widget can't do directly". The expanded scope is documented in `docs/setup-conventions.md` §7.

## Known issues / follow-ups (v0.2.x)

- Document the main-glob requirement in AGENTS.md so future agentic workers don't repeat the 404 mistake when adding a third function.
- The Function App Platform CORS preflight rejection is a real quirk; if we ever need to allow custom request headers from widget POSTs, we'll need a different workaround (maybe disable Platform CORS via Azure CLI app-setting, or handle preflight differently in code).
